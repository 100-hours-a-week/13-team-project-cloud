# Contributing Guide

## Issue 관리

### Issue 템플릿

| 템플릿 | 용도 | 기본 라벨 | 자동 라벨링 |
|--------|------|----------|------------|
| Sprint Backlog | 스프린트 백로그 항목 | `백로그` | 우선순위 |
| Bug Report | 버그/오류 리포트 | `버그` | 심각도 |
| Incident Report | 장애/인시던트 기록 | `장애` | 심각도 |
| Feature Request | 기능 요청/개선 제안 | `기능요청` | 우선순위 |
| Question | 질문/논의 | `질문` | - |

### 라벨

#### 기본 라벨

| 라벨 | 설명 |
|------|------|
| `백로그` | 스프린트 백로그 항목 - 주간 과제 및 작업 단위 |
| `버그` | 버그/오류 - 기능이 의도대로 동작하지 않는 문제 |
| `장애` | 서비스 장애/인시던트 - 긴급 대응 필요한 운영 이슈 |
| `기능요청` | 기능 요청/개선 - 새로운 기능 추가 또는 기존 기능 개선 제안 |
| `질문` | 질문 및 논의 - 기술적 질문이나 의사결정 필요한 사항 |

#### 우선순위 라벨 (자동)

Sprint Backlog, Feature Request 템플릿에서 우선순위 선택 시 자동 적용

| 라벨 | 설명 |
|------|------|
| `우선순위:높음` | 긴급 - 즉시 처리 필요, 다른 작업보다 먼저 수행 |
| `우선순위:중간` | 보통 - 이번 스프린트 내 처리 권장 |
| `우선순위:낮음` | 여유 - 시간 여유 있을 때 처리해도 무방 |

#### 심각도 라벨 (자동)

Bug Report, Incident Report 템플릿에서 심각도 선택 시 자동 적용

| 라벨 | 설명 |
|------|------|
| `심각도:Critical` | 서비스 전체 중단 |
| `심각도:High` | 주요 기능 장애 |
| `심각도:Medium` | 일부 기능 장애 |
| `심각도:Low` | 경미한 영향 |

### Milestone (스프린트 관리)

주차별 Milestone을 사용하여 스프린트를 관리합니다.

1. **Milestone 생성**: `Week 1`, `Week 2` 등 주차별로 생성
2. **Issue 연결**: 백로그 이슈 생성 시 해당 주차 Milestone 선택
3. **진행률 확인**: Milestone 페이지에서 전체 진행률 확인

### Issue 작성 규칙

1. 적절한 템플릿 선택
2. 제목은 명확하게 작성
3. 필수 항목 모두 작성
4. 해당하는 Milestone 연결
5. 담당자(Assignee) 지정

## 커밋 컨벤션

### 커밋 메시지 형식

```
<type>: <subject>

[body]
```

### 커밋 타입

| 타입 | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 추가 | `feat: EC2 인스턴스 생성 스크립트 추가` |
| `fix` | 버그 수정 | `fix: 보안그룹 포트 설정 오류 수정` |
| `docs` | 문서 변경 | `docs: README 배포 가이드 추가` |
| `chore` | 설정, 빌드, 패키지 등 | `chore: GitHub Actions 워크플로우 추가` |
| `refactor` | 코드 리팩토링 | `refactor: 배포 스크립트 구조 개선` |
| `test` | 테스트 추가/수정 | `test: 배포 스크립트 테스트 추가` |
| `style` | 코드 포맷팅, 세미콜론 등 | `style: 코드 포맷팅 적용` |

### 커밋 작성 규칙

1. **제목은 50자 이내**로 작성
2. **제목 끝에 마침표 사용하지 않음**
3. **명령문으로 작성** (added X → add O)
4. **본문은 선택사항**, 필요 시 상세 내용 작성
5. **하나의 커밋은 하나의 작업**만 포함

### 예시

```
feat: S3 버킷 생성 Terraform 코드 추가

- 정적 웹 호스팅용 버킷 설정
- 버전 관리 활성화
- 관련 이슈: #12
```

```
fix: RDS 보안그룹 인바운드 규칙 수정

잘못된 포트 번호(3000 → 3306)로 인해
DB 연결이 실패하던 문제 해결
```

## 브랜치 전략

GitHub Flow 기반의 간단한 브랜치 전략을 사용합니다.

### 브랜치 구조

```
main
├── feature/*   ← 기능/작업 브랜치
└── hotfix/*    ← 긴급 수정 브랜치
```

### 브랜치 설명

| 브랜치 | 용도 | 예시 |
|--------|------|------|
| `main` | 배포 가능한 안정 상태 유지 | - |
| `feature/*` | 새 기능, 작업, 개선 | `feature/ec2-setup`, `feature/cicd-pipeline` |
| `hotfix/*` | 긴급 버그 수정 | `hotfix/security-patch` |

### 작업 흐름

1. **브랜치 생성**: `main`에서 `feature/작업명` 브랜치 생성
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/작업명
   ```

2. **작업 및 커밋**: 커밋 컨벤션에 따라 작업
   ```bash
   git add .
   git commit -m "feat: 기능 설명"
   ```

3. **Push 및 PR 생성**
   ```bash
   git push origin feature/작업명
   ```
   - GitHub에서 PR 생성
   - PR 제목도 커밋 컨벤션 따르기 (예: `feat: EC2 인스턴스 설정 추가`)

4. **코드 리뷰 후 머지**
   - 최소 1명 이상 리뷰 권장
   - 머지 후 feature 브랜치 삭제

### 긴급 수정 (Hotfix)

1. `main`에서 `hotfix/설명` 브랜치 생성
2. 수정 후 PR 생성
3. 빠른 리뷰 후 `main`에 머지

### 주의사항

- `main` 브랜치에 직접 push 금지
- PR 없이 머지 금지
- 충돌 발생 시 로컬에서 해결 후 push
